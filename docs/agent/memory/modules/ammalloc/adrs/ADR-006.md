# ADR-006: HugePageCache 无锁双栈架构

> 记录 HugePageCache 从 `std::mutex` 到零分配无锁双栈（Lock-Free Dual-Stack）的架构演进决策。

- 标题：`HugePageCache 无锁双栈架构`
- 状态：`accepted`
- 作用范围：
  - 影响模块：`ammalloc`
  - 影响子模块：`page_allocator`
  - 关联记忆：
    - `docs/agent/memory/modules/ammalloc/module.md`
    - `docs/agent/memory/modules/ammalloc/submodules/page_allocator.md`
    - `docs/designs/ammalloc/page_allocator_design.md`

## 背景

- **问题识别**：16 线程并发测试显示，基于 `std::mutex` 的 `HugePageCache` 吞吐量从单线程 4.3M/s 骤降至 638k/s，性能损失达 **4.8x**。
- **根本原因**：所有线程在 `Get()` / `Put()` 操作上都串行竞争同一把 `std::mutex`，形成严重瓶颈。
- **约束条件**：
  - 必须遵守 ammalloc 的**零分配自举约束**（Zero-Allocation Bootstrapping）：核心路径禁止 `new/delete`、`std::vector` 等可能触发递归 malloc 的操作。
  - 必须保证线程安全，正确处理 ABA 问题。
  - 必须保持 LIFO（后进先出）语义以优化缓存局部性。

## 决策

采用**零分配无锁双栈（Lock-Free Dual-Stack）**架构替换原有基于 `std::mutex` 的实现：

1. **双栈结构**：维护两个无锁栈 `free_head_`（空闲槽位栈）和 `used_head_`（已占用槽位栈）。
2. **固定数组**：使用 `Slot slots_[kCapacity]` 固定数组，完全零分配。
3. **ABA 防护**：将 16-bit 索引和 48-bit Tag 打包进单个 `std::atomic<uint64_t>`，CAS 操作时 Tag 递增，避免 ABA 问题。
4. **数据竞争防护**：`Slot::next` 使用 `std::atomic<uint16_t>`，防止并发回收时的数据竞争。
5. **缓存对齐**：`free_head_` 和 `used_head_` 按 `CACHE_LINE_SIZE` 对齐，避免伪共享。

## 原因

### 为什么采用该方案

- **性能提升**：16 线程吞吐量从 **1.07 M/s** 提升至 **2.24-2.40 M/s**，提升 **110-124%**。
- **零分配合规**：完全使用固定数组，无 `new/delete`，符合 ammalloc 自举约束。
- **可扩展性**：无锁设计消除了单点瓶颈，线程数增加时性能下降显著减缓。
- **ABA 安全**：48-bit Tag 提供足够的版本空间（$2^{48}$ 次操作），实际不可能回绕。

### 为什么不继续沿用旧方案

- **性能不可接受**：`std::mutex` 在 16 线程下造成 4.8x 性能损失，无法满足高并发推理引擎的需求。
- **无法优化**：`std::mutex` 的串行特性无法通过代码优化消除，必须改变同步原语。

### 为什么不是其他方案

- **方案 A：无锁链表（Tagged Pointer）**：需要动态分配节点（`new Node`），违反零分配约束。
- **方案 B：线程本地缓存（TLS）**：虽然减少竞争，但 `thread_local` 初始化可能触发 malloc，且实现复杂。
- **方案 C：分片缓存（Sharded Cache）**：需要多个锁或无锁结构，内存占用增加，收益不如完全无锁。

## 影响

### 接口
- **无变化**：`HugePageCache::GetInstance()`、`Get()`、`Put()`、`ReleaseAllForTesting()` 接口保持不变，调用方无需修改。

### 架构
- **新增辅助方法**：`PageAllocator::RecordMunmapFailure()` 用于缓存内部统计 `munmap` 失败。
- **头文件更新**：`ammalloc/include/ammalloc/page_allocator.h` 新增辅助方法声明。

### 并发
- **同步原语变更**：从 `std::mutex` 改为基于 `std::atomic` 的 CAS 循环。
- **内存序策略**：`Pop` 使用 `acquire/acquire`，`Push` 使用 `release/relaxed`。

### 性能
- **显著提升**：16 线程吞吐量提升 110-124%，单线程性能持平。
- **缓存行为**：缓存行对齐避免伪共享，CAS 自旋在轻竞争下开销极低。

### 验证
- **单元测试**：`PageAllocatorTest.*`、`PageAllocatorThreadSafeTest.*` 15/15 通过。
- **基准测试**：`BM_PageAllocator_AllocFree_2M_MultiThreadContention` 16 线程达到 2.24-2.40 M/s。
- **并发压力测试**：Oracle 交叉审查确认无数据竞争和 ABA 漏洞。

### 迁移
- **向后兼容**：接口完全兼容，无需调用方修改代码。
- **行为一致**：保持 LIFO 语义，不影响缓存命中率。
