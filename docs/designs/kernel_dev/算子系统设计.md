## 1. 架构设计哲学 (Core Philosophy)

在现代大模型推理引擎中，极致的低延迟（Low Latency）和高吞吐（High Throughput）是第一要务。为此，我们的算子系统严格遵循以下核心哲学：

1. **“运行时零开销 (Zero-overhead during Run)”**：所有的类型推导、Shape 检查、硬件特性（CPU Flags）探测，**必须全部在模型加载（Prepare）阶段完成**。前向推理 Run 阶段禁止 dtype dispatch、shape 推导、硬件能力探测、kernel 查找、动态分配等高成本逻辑；允许保留极少量不可避免的轻量控制流，例如函数指针调用、debug-only check、profiling guard、错误返回传播。
2. **“运行时零分配 (Steady-state Zero Allocation)”**：所有算子所需的临时内存（Workspace）必须在初始化阶段计算完毕，由底层内存池统一分配。`Run` 阶段严禁调用 `malloc/new`。
3. **“严格的分层隔离 (Separation of Concerns)”**：图逻辑、多线程调度逻辑、底层硬件计算指令必须严格拆分，做到高内聚、低耦合。

基于上述哲学，本系统采用**标准的“三层架构（3-Tier Architecture）”**。

## 2. 核心三层架构剖析 (Architecture Breakdown)

每一个算子的完整生命周期，必须严格切分为以下三个独立层级：

### 🐟 第一层：图节点层 (Graph Op Layer)
* **角色定位**：算子元数据档案袋与 AOT 路由器。
* **核心职责**：
  * **前期准备 (Prepare)**：检查参数合法性、推导 Output Shape、计算 Workspace 需求大小。
  * **编译期路由 (AOT Dispatch)**：根据 Tensor 类型（FP32/FP16）和硬件特征（AVX2/AVX512），**直接将执行期函数指针绑定到对应的第二层调度函数上**。
  * **执行入口 (Run)**：仅做一层极薄的函数指针调用，将上下文直接轰给下层。
* **代码特征**：面向对象（OOP），继承自 `Operator` 基类。

### 🦈 第二层：宏观调度层 (Macro-Kernel Layer)
* **角色定位**：数据解包与任务切分。
* **核心职责**：
  * **数据解包**：从框架庞大的 `KernelInvocation` 对象中提取出底层的裸指针（`float*`）、形状（Shape）和超参。
  * **多核调度**：决定任务的横向切分策略，调用 `ThreadPool` 进行多线程并发（如将 `Batch` 或 `SeqLen` 维度的任务均分给各物理核）。
  * **地址计算**：处理内存步长（Stride），计算每个线程具体处理的内存偏移量。
* **代码特征**：包含 `for` 循环、线程池操作，但**绝对没有**具体的硬件计算指令。

### 🐋 第三层：微内核层 (Micro-Kernel Layer)
* **角色定位**：一线流水线、纯粹的硅片压榨。
* **核心职责**：
  * **极致计算**：直接操作寄存器，执行 AVX/NEON/CUDA 等底层指令（如 FMA 乘加、VNNI 整数点积）。
  * **微观优化**：负责指令级并行（ILP）、循环展开（Loop Unrolling）、缓存重排（Cache Blocking）和非对齐尾部（Tail）处理。
* **代码特征**：函数签名极其扁平，只认裸指针，强依赖 `__restrict__` 关键字，没有任何框架头文件的依赖。

## 3. 代码范例：以 RMSNorm 为例

以下代码展示了如何利用三层架构实现一个高性能的 `RMSNorm` 算子。

### Layer 1: RmsNormOp (图节点层)
```cpp
// rmsnorm_op.h
class RmsNormOp final : public Operator {
public:
    struct Params {
        float epsilon_ = 1.0e-5F;
    };

    explicit RmsNormOp(Params params) noexcept : params_(params) {}

    AM_NODISCARD OpType Type() const noexcept override { return OpType::kRmsNorm; }
    AM_NODISCARD const char* Name() const noexcept override { return "RmsNorm"; }

    // 核心准备逻辑：AOT 路由，锁定底层实现
    AM_NODISCARD Status Prepare(OperatorContext& ctx) override {
        auto dtype = ctx.GetInputDtype(0);
        auto features = ctx.GetHardwareFeatures();

        // 编译期静态路由：无脑指向最契合的 Macro-Kernel
        if (dtype == DataType::kFloat32 && features.has_avx2) {
            resolved_kernel_.fn = dispatch_rmsnorm_fp32_avx2;
        } else if (dtype == DataType::kFloat32) {
            resolved_kernel_.fn = dispatch_rmsnorm_fp32_scalar;
        } else {
            return Status(StatusCode::kUnimplemented, "Unsupported dtype or hardware");
        }
        return Status::OK();
    }

    // 核心执行逻辑：零检查
    AM_NODISCARD Status Run(const KernelInvocation& invocation,
                            const KernelContext& op_ctx,
                            const WorkspaceBinding& workspace) const noexcept override {
        // 直接执行在 Prepare 阶段绑定的函数指针
        return resolved_kernel_.fn(invocation, op_ctx, workspace);
    }

private:
    Params params_{};
    ResolvedKernel resolved_kernel_{};
};
```

### Layer 2: 宏观调度层 (Macro-Kernel)
```cpp
// rmsnorm_dispatch.cpp
// 注意：签名必须与 resolved_kernel_.fn 严格匹配
Status dispatch_rmsnorm_fp32_avx2(const KernelInvocation& inv, 
                                  const KernelContext& ctx, 
                                  const WorkspaceBinding& ws) 
{
    // 1. 解包底层裸指针与维度
    const float* in = inv.GetInputTensor(0).Data<float>();
    const float* w  = inv.GetInputTensor(1).Data<float>();
    float* out      = inv.GetOutputTensor(0).Data<float>();
    
    const int num_tokens  = inv.GetInputShape(0).GetRows();
    const int hidden_size = inv.GetInputShape(0).GetCols();
    const float eps       = ctx.GetAttribute<float>("epsilon");

    // 2. 任务切分与多核分发
    ctx.GetThreadPool()->parallel_for(0, num_tokens, [&](int start, int end) {
        for (int i = start; i < end; ++i) {
            // 3. 计算指针偏移，调用微内核
            rmsnorm_micro_kernel_avx2(
                out + i * hidden_size, 
                in + i * hidden_size, 
                w, 
                hidden_size, 
                eps
            );
        }
    });

    return Status::OK();
}
```

## 4. 算子开发规约 (Developer Guidelines)

所有参与底层算子开发的工程师，必须严格遵守以下契约：

1. **微内核的 `__restrict__` 誓言**：
   在 Layer 3 的签名中，所有的输出指针必须标记为 `__restrict__`（或 MSVC 的 `__restrict`）。这授权了编译器进行最激进的向量化优化，禁止传入有内存重叠（Aliasing）的 Tensor，如有 In-place 需求，必须在 Layer 2 提前处理。
2. **微内核的纯净性 (Purity)**：
   Layer 3 的代码文件严禁 `#include` 任何上层图节点的头文件（如 `tensor.h`, `operator.h`）。它应当纯净到可以直接被复制粘贴到其他 C 语言项目中编译运行。
3. **返回值的严谨性 (AM_NODISCARD)**：
   Layer 1 和 Layer 2 的所有函数必须使用 `[[nodiscard]]` (或 `AM_NODISCARD` 宏) 修饰，强制调用方处理 `Status` 错误码，严禁静默吞没错误。
4. **异常安全 (noexcept)**：
   推理引擎的主执行链路（`Run` 以及其调用的 `dispatch` 函数）应当标记为 `noexcept`。使用 `Status` 替代 C++ Exception 机制，保证极低的调用栈解退（Stack Unwinding）开销。
5. **严禁运行时判空**：
   输入 Tensor 是否为空、Shape 是否合法等防御性编程，**仅允许出现在 `Prepare` / `CheckShapes` 函数中**。`Run` 函数默认所有内存地址绝对合法，直接进行读写，以此换取极限性能。
