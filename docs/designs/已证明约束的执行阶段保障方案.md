# 形状约束证明与执行期保障方案

## 一、问题定义

当前形状约束系统需要同时解决两个问题：

1. **Plan-time 约束证明**：哪些 shape requirement 可以在构建执行计划时被证明，从而不进入 runtime checks。
2. **Runtime 执行期保障**：被 plan-time 证明并裁剪掉的约束，在执行阶段如何保证其证明前提没有漂移。

形状约束仍采用三态模型：

| 评估结果 | 含义 | 处理方式 |
| --- | --- | --- |
| `kSatisfied` | plan-time 已证明 | 不进入 `runtime_checks`，由 TensorSpec runtime validation 保障证明前提 |
| `kViolated` | plan-time 已发现违规 | 构建失败 |
| `kDeferred` | plan-time 无法确定 | 作为 `runtime_checks` 在执行阶段验证 |

核心风险是：`runtime_checks` 只覆盖 `kDeferred` 约束。如果一个约束在 plan-time 被判定为 `kSatisfied`，它不会进入 `runtime_checks`。此时执行阶段必须验证该证明依赖的 TensorSpec 前提仍然成立，否则同一个 `ExecutionPlan` 被绑定到不兼容 tensor 时，已证明约束可能被破坏。

典型例子：

```text
Plan-time:
  input0: [S, H]
  input1: [H]
  => input0.dim1 == input1.dim0 已由同一个 ShapeSymbol(H) 证明

Runtime:
  input0: [2, 8]
  input1: [16]
```

如果执行阶段只检查 `runtime_checks`，这个错误不会被发现，因为该约束已在 plan-time 被裁剪。因此执行阶段需要两类验证：

```text
TensorSpec runtime validation
  -> 验证 kSatisfied 约束依赖的 dtype/rank/static dim/symbol identity 前提没有漂移

ShapeConstraint runtime validation
  -> 验证 kDeferred 约束在 concrete shape 下成立
```

## 二、核心语义

### 2.1 Proven fact 与 requirement

`SymbolConstraintSolver` 只能保存 **已证明事实（proven fact）**，不能保存普通 operator 的待验证要求（requirement）。

已证明事实示例：

- 同一个 `ShapeSymbol::value()` 表示的符号 identity。
- 符号已被可信上游逻辑绑定到某个 static dimension。
- 未来图级 shape environment 明确证明两个端口维度来自同一 tensor dimension。

待验证约束示例：

- RmsNorm 要求 `input.hidden == weight.len`。
- Broadcast 要求两个维度满足 `a == b || a == 1 || b == 1`。
- Reshape 要求输入输出 volume 相等。

待验证约束不能直接写入 solver。它们必须先通过 solver 尝试证明；若无法证明但可在 runtime 检查，则保留为 `runtime_checks`。

### 2.2 为什么不能把 requirement 写入 solver

以 RmsNorm 为例：

```text
input.shape  = [S2, S3]
weight.shape = [S4]
requirement  = S3 == S4
```

`S3` 与 `S4` 是不同符号，只能说明 operator 要求它们相等，不能说明 plan-time 已经证明它们相等。如果先执行：

```cpp
solver.AddEqual(S3, S4);
```

再用 solver 评估同一个 `DimEqualConstraint`，结果会变成 `kSatisfied`，从而错误删除 runtime check。正确行为是：solver 无法证明 `S3 == S4` 时，保留该约束到 `runtime_checks`。

### 2.3 `ShapeSymbol` runtime 语义

| `ShapeSymbol` 类型 | Plan-time 含义 | Runtime validation 行为 |
| --- | --- | --- |
| static dim `>= 0` | 已知静态维度 | concrete dim 必须等于 static value |
| generated symbol `<= -2` | 具有 identity 的动态维度 | 同一次 `Execute()` 中必须绑定到同一个 concrete dim |
| `Unknown(-1)` | 无 identity 的未知维度 | 不绑定、不传播，只要求 concrete dim 合法 |

`Unknown(-1)` 不能进入 solver 的 equality class，也不能进入 runtime binding map。

## 三、实施顺序与总体数据流

本方案分阶段实施。关键原则是：**runtime TensorSpec validation 可以先做；完整 plan-time solver proof 应等最小 ModelGraph / ValueId / ShapeEnvironment 建立后再做**。

推荐顺序：

```text
Phase 1A：Runtime TensorSpec validation
    不依赖 ModelGraph，先补齐 kSatisfied 约束的执行期前提验证。

Phase 1B：最小 ModelGraph / ValueId / ShapeEnvironment
    建立可信 graph value identity，作为 solver proven facts 的来源。

Phase 1C：SymbolConstraintSolver plan-time proof
    基于 ModelGraph / ShapeEnvironment 生成 proven facts，并裁剪 runtime checks。
```

完整目标数据流如下：

```text
ModelGraph / GraphCompiler
    │
    ├── 维护 graph values / edges / TensorSpec / ShapeEnvironment
    │
    ├── 创建 plan-time ShapeInferenceContext
    │       └── SymbolConstraintSolver solver
    │
    ├── Operator::CheckInputSpecs(value input_specs)
    │       └── 只做 arity / dtype / rank / 静态硬错误检查
    │
    ├── Operator::InferOutputShapes(input_specs, context)
    │       ├── 产出 outputs
    │       └── 产出待验证 ShapeConstraint 列表
    │
    ├── ProveShapeConstraints(context, constraints, inputs, outputs)
    │       ├── 已证明 satisfied：删除 runtime check
    │       ├── 已证明 violated：plan-time error
    │       └── 无法证明但 runtime 可检查：保留 runtime check
    │
    └── ExecutionStep
            ├── input_specs
            ├── output_specs
            └── runtime_checks only

LayerRunner::Run(plan, bindings)
    │
    ├── 创建 per-Execute ShapeBindingContext
    │
    └── RunStep(...)
            ├── ValidateAndBind input_specs
            ├── ValidateAndBind output_specs
            ├── ValidateRuntimeShapeConstraints(runtime_checks)
            └── op->Run()
```

## 四、Plan-time proof 设计

### 4.1 ShapeInferenceContext

`ShapeInferenceContext` 属于 Phase 1C。它应在最小 `ModelGraph` / `ShapeEnvironment` 建立后引入，由 graph compiler 或升级后的 `ExecutionPlanBuilder` 持有。

在 Phase 1A 中，不需要 `ShapeInferenceContext`；runtime TensorSpec validation 只依赖 `ExecutionStep.input_specs` / `output_specs`。

在 Phase 1B/1C 中，context 可以逐步扩展为 graph-aware plan-time 状态容器。

```cpp
class SymbolConstraintSolver;

struct ShapeInferenceContext {
    SymbolConstraintSolver& solver;
    // Phase 1C+ may add ShapeEnvironment, ModelGraph, NodeId, diagnostics, etc.
};
```

`ShapeInferenceContext` 不进入 `ExecutionStep`，不进入 runtime plan。

### 4.2 InferenceResult 保持轻量

`InferenceResult` 不持有 solver。

```cpp
struct InferenceResult {
    std::vector<TensorSpec> outputs{};
    std::vector<ShapeConstraint> runtime_checks{};
};
```

`runtime_checks` 在 operator 返回时表示“operator 产生的待验证约束”。经过 `ProveShapeConstraints` 后，只有未被 plan-time 证明的约束才进入 `ExecutionStep::runtime_checks`。

### 4.3 Operator::InferOutputShapes 增加 context

旧接口：

```cpp
virtual StatusOr<InferenceResult> InferOutputShapes(
        std::span<const TensorSpec> inputs) const = 0;
```

新接口：

```cpp
virtual StatusOr<InferenceResult> InferOutputShapes(
        std::span<const TensorSpec> inputs,
        ShapeInferenceContext& context) const = 0;
```

约束：

- `InferOutputShapes` 可以读取 `context.solver` 来消除局部不确定性。
- `InferOutputShapes` 只有在事实确实已被证明时才可写 solver。
- 普通 operator compatibility requirement 应作为 `ShapeConstraint` 返回，而不是写入 solver。

### 4.4 ShapeConstraint proof API

移除 public `EvaluateShapeConstraint` 的 symbolic/runtime overload 语义，改为两个明确命名的路径。

```cpp
struct ShapeConstraintProofContext {
    const SymbolConstraintSolver& solver;
};

struct ShapeConstraintProofResult {
    ShapeConstraintList runtime_checks{};
};

AM_NODISCARD StatusOr<ShapeConstraintProofResult> ProveShapeConstraints(
        ShapeConstraintProofContext context,
        std::span<const ShapeConstraint> constraints,
        std::span<const SymbolicShape> inputs,
        std::span<const SymbolicShape> outputs);

AM_NODISCARD Status ValidateRuntimeShapeConstraints(
        std::span<const ShapeConstraint> constraints,
        std::span<const TensorView> inputs,
        std::span<const MutableTensorView> outputs);
```

`ProveShapeConstraints` 的语义：

| Proof 结果 | 行为 |
| --- | --- |
| satisfied | 丢弃该 constraint，不生成 runtime check |
| violated | 返回 `Status::InvalidArgument` |
| deferred | 加入 `ShapeConstraintProofResult::runtime_checks` |

`ValidateRuntimeShapeConstraints` 的语义：

- concrete check 通过：继续。
- concrete check 失败：返回 `Status::InvalidArgument`。
- port/dim 引用错误：返回对应 `Status`。
- 不暴露 `kDeferred`，运行时不应存在 deferred。

## 五、UnifyShapeSymbol 改造

### 5.1 不把 symbolic collision 写入 solver

`UnifyShapeSymbol` 仍用于“合并两个 shape 表示”，不能把普通 equality requirement 伪装成 proven fact。

推荐第一阶段提供只读 solver 版本：

```cpp
StatusOr<ShapeSymbol> UnifyShapeSymbol(const ShapeSymbol& a,
                                       const ShapeSymbol& b,
                                       const SymbolConstraintSolver& solver);
```

语义：

1. `a == b`：返回 `a`。
2. static/static 不同：返回 `InvalidArgument`。
3. static/symbolic：如果 solver 证明 symbolic 绑定到同一 static，返回 static；如果绑定到不同 static，返回 `InvalidArgument`；否则返回 static 仅表示输出 shape 可采用更具体的静态维度，**不隐含该 equality requirement 已被证明**。调用方仍必须生成对应 `ShapeConstraint`，由 `ProveShapeConstraints` 决定是否删除或保留 runtime check。
4. unknown：返回另一个 symbol。
5. symbolic/symbolic：
   - solver 已证明相等：返回其中一个代表。
   - solver 已证明静态绑定相同：返回该 static value。
   - solver 已证明静态绑定冲突：返回 `InvalidArgument`。
   - 无法证明：返回 `ShapeSymbol::Unknown()`，不写 solver。

### 5.2 SymbolicShape::Unify 增加 solver 参数

```cpp
StatusOr<SymbolicShape> SymbolicShape::Unify(
        const SymbolicShape& other,
        const SymbolConstraintSolver& solver) const;
```

`Join` 不需要 solver。`Join` 用于控制流分支放宽，遇到分歧时生成 fresh symbol，而不是记录 equality。

## 六、Constraint proof 规则

### 6.1 DimEqualConstraint

步骤：

1. 解析 lhs/rhs 维度。
2. 任一 shape unranked：deferred。
3. 调用 `solver.EvaluateEqual(lhs, rhs)`。

结果：

| solver 结果 | proof 行为 |
| --- | --- |
| `kSatisfied` | 删除 runtime check |
| `kViolated` | plan-time error |
| `kDeferred` | 保留 runtime check |

### 6.2 DimBroadcastableConstraint

步骤：

1. 如果 `solver.AreEqual(lhs, rhs)`：satisfied。
2. 通过 `solver.GetStaticBinding(lhs/rhs)` 获取可证明的静态值。
3. 如果任一可证明值为 1：satisfied。
4. 如果两边都有可证明静态值但不满足 broadcast：violated。
5. 否则 deferred。

注意：`GetStaticBinding` 对 static dim 会直接返回 static value，因此不需要额外分支处理 static/static。

### 6.3 VolumeEqualConstraint

第一阶段只支持保守证明：

1. **逐位 proven equal fast-path**

   如果 lhs/rhs 维度列表长度相同，并且每一对维度都满足 `solver.AreEqual(lhs_dim, rhs_dim)`，则 satisfied。

2. **静态绑定参与 volume 计算**

   计算每侧 volume 时：

   - static dim：使用 static value。
   - symbolic dim 且 solver 有 static binding：使用 binding。
   - unknown / 无 binding symbolic：deferred。

3. 两侧均可计算时比较 product。

不做：

- 不证明乘法交换律。
- 不做部分约分。
- 不把 volume equality 反向写入 solver。

### 6.4 RankEqualConstraint / RankAtLeastConstraint

rank 当前不是 `ShapeSymbol`，solver 不参与。

| rank 状态 | proof 行为 |
| --- | --- |
| rank 已知且满足 | satisfied |
| rank 已知但不满足 | violated |
| unranked | deferred |

## 七、ExecutionPlanBuilder / ModelGraph 集成

### 7.1 Phase 1A：ExecutionPlanBuilder 只保存 runtime validation 所需规格

在没有 `ModelGraph` 前，`ExecutionPlanBuilder` 不应尝试推导跨 step solver 等价类。Phase 1A 只做不依赖图结构的改造：把构建时的 input/output specs 保存到 `ExecutionStep`，供 runtime TensorSpec validation 使用。

```cpp
ExecutionStep{
    .input_specs = node.input_specs,
    .output_specs = std::move(inferred->outputs),
    .runtime_checks = std::move(inferred->runtime_checks),
};
```

该阶段可以继续保守保留 operator 产生的 `runtime_checks`，不做 solver proof 裁剪。

### 7.2 Phase 1B：最小 ModelGraph / ValueId / ShapeEnvironment

正式接入 solver proof 前，应先建立最小图级 IR：

```text
ModelGraph
  - NodeId
  - ValueId
  - node inputs / outputs
  - graph inputs / outputs
  - producer-consumer edges
  - value -> TensorSpec / shape facts
  - topological order
```

`ValueId` 是 solver proven facts 的主要来源。同一个 graph value 的同一维度可以复用同一个 `ShapeSymbol::value()`，或者由 `ShapeEnvironment` 明确写入 solver。

### 7.3 Phase 1C：基于 graph identity 的 solver proof

有了 `ModelGraph` 后，graph compiler 或升级后的 `ExecutionPlanBuilder` 创建 plan-time solver/context：

```cpp
SymbolConstraintSolver solver;
ShapeInferenceContext inference_context{.solver = solver};
```

每个 registered operator inference 使用 graph-aware context：

```cpp
auto inferred = op->InferOutputShapes(value_input_specs, inference_context);
```

随后对 operator 返回的 constraints 做 proof：

```cpp
auto proof = ProveShapeConstraints(
        ShapeConstraintProofContext{.solver = solver},
        inferred->runtime_checks,
        ExtractSymbolicShapes(value_input_specs),
        ExtractSymbolicShapes(inferred->outputs));
```

最终进入 step 的只有 proof 后仍 deferred 的 runtime checks：

```cpp
ExecutionStep{
    .input_specs = value_input_specs,
    .output_specs = std::move(inferred->outputs),
    .runtime_checks = std::move(proof->runtime_checks),
};
```

### 7.4 ExecutionStep 保存 TensorSpec，不保存 solver

`ExecutionStep` 需要保存 input/output specs 以支持 runtime TensorSpec validation，但不保存 solver：

```cpp
struct ExecutionStep {
    KernelSelector selector{};
    OperatorPtr op{};
    const void* packed_weights = nullptr;
    WorkspaceRequirement workspace_requirement{};
    std::vector<TensorSpec> input_specs{};
    std::vector<TensorSpec> output_specs{};
    std::vector<ShapeConstraint> runtime_checks{};
    const char* debug_name = nullptr;
};
```

理由：

- solver 是 plan-time proof 工具。
- LayerRunner 不需要也不应该读取 solver。
- input/output specs 是 runtime validation 的依据。
- 避免引入运行时符号绑定语义。

### 7.5 CheckInputSpecs 职责收窄

builder 调用顺序保持：

```text
ValidateParams → CheckInputSpecs → InferOutputShapes → Prepare
```

`CheckInputSpecs` 不接 solver，因此只做：

- 输入数量。
- dtype。
- rank。
- static-only 硬错误。
- 静态维度合法性，例如 positive if static。

symbolic equality / broadcast / volume requirement 应在 `InferOutputShapes` 中生成 `ShapeConstraint`，再交给 `ProveShapeConstraints` 裁剪。

## 八、Runtime TensorSpec validation

### 8.1 ShapeBindingContext

执行阶段直接使用 `TensorSpec` 作为验证输入，不新增 fingerprint 类型。唯一需要新增的是 per-Execute 局部绑定表：

```cpp
struct ShapeBindingSource {
    size_t step_index;
    TensorPort port;
    size_t dim_index;
};

struct ShapeBinding {
    int64_t value;
    ShapeBindingSource source;
};

class ShapeBindingContext {
public:
    Status Bind(ShapeSymbol symbol,
                int64_t value,
                ShapeBindingSource source);

    StatusOr<int64_t> Lookup(ShapeSymbol symbol) const;

private:
    std::unordered_map<int64_t, ShapeBinding> bindings_;
};
```

`ShapeBindingContext` 不应暴露为新的图推导入口。它只是 runtime TensorSpec validator 的内部状态，用来保证同一个 symbol identity 在执行阶段没有绑定冲突。

### 8.2 ValidateAndBindTensorSpec

建议 API：

```cpp
Status ValidateAndBindTensorSpec(const TensorSpec& spec,
                                 const TensorView& tensor,
                                 ShapeBindingContext& bindings,
                                 ShapeBindingSource source);

Status ValidateAndBindTensorSpec(const TensorSpec& spec,
                                 const MutableTensorView& tensor,
                                 ShapeBindingContext& bindings,
                                 ShapeBindingSource source);
```

验证内容：

- dtype 必须一致。
- rank 必须一致。
- static dim 必须等于 concrete dim。
- generated symbol 第一次绑定记录 concrete dim，后续绑定必须一致。
- `Unknown(-1)` 不绑定、不传播 identity。

伪代码：

```cpp
Status ShapeBindingContext::Bind(ShapeSymbol symbol,
                                 int64_t value,
                                 ShapeBindingSource source) {
    if (value < 0) {
        return Status::InvalidArgument("runtime dimension must be non-negative");
    }

    if (symbol.IsStatic()) {
        if (symbol.GetStaticValue() != value) {
            return Status::InvalidArgument("static shape dimension mismatch");
        }
        return Status::Ok();
    }

    if (symbol.IsUnknown()) {
        return Status::Ok();
    }

    auto [it, inserted] = bindings_.emplace(symbol.value(), ShapeBinding{value, source});
    if (!inserted && it->second.value != value) {
        return Status::InvalidArgument("symbolic shape binding conflict");
    }
    return Status::Ok();
}
```

### 8.3 LayerRunner 执行顺序

`LayerRunner::Run()` 为每次执行创建一个 `ShapeBindingContext`，并把它传给每个 step。这个 context 是 per-Execute 的，因此会跨 step 共享：

```text
LayerRunner::Run(plan, bindings):
  ShapeBindingContext shape_bindings;
  for each step:
    RunStep(step_index, step, bindings, shape_bindings)
```

`RunStep()` 中的顺序应使用 `ValidateAndBind` 语义：

```text
1. Bind workspace
2. 读取 StepTensorBinding 中的 inputs / outputs
3. ValidateAndBind input_specs against TensorView inputs
4. ValidateAndBind output_specs against MutableTensorView outputs（如果输出已绑定）
5. ValidateRuntimeShapeConstraints(runtime_checks)
6. op->Run()
```

如果 Step 1 的 input tensor 与 Step 0 的 output tensor 是同一个实际 tensor，该机制会自然验证一致性。如果它们不是同一个 tensor，但 `TensorSpec` 引用了同一个 `ShapeSymbol`，也会强制它们的 concrete dim 一致；这是正确行为，因为同一个 generated `ShapeSymbol` 本身就表达 runtime identity 约束。

### 8.4 与 runtime_checks 的职责边界

两者是互补关系：

| 组件 | 负责内容 | 示例 |
| --- | --- | --- |
| TensorSpec runtime validator | 符号阶段已证明约束的前提 | 同一个 `H` 在多个 input dim 上必须绑定为相同值 |
| ShapeConstraint runtime validator | 符号阶段无法证明的 deferred 约束 | 两个不同 symbol 在 runtime 必须相等 |

`runtime_checks` 不应重新检查所有已证明约束，否则会重复建模并引入两套 truth source。已证明约束由 TensorSpec validator 保障，deferred 约束由 `ShapeConstraint` evaluator 保障。

## 九、跨 step 语义

`ShapeBindingContext` 可以跨 step 传播 symbol binding，但它有一个关键前提：不同 step 的 `TensorSpec` 必须让相同语义维度引用同一个 `ShapeSymbol::value()`。

```text
Step 0 output_specs: output0 = [B, S]
Step 0 runtime output0: [1, 4096]
  -> bind S = 4096

Step 1 input_specs: input0 = [B, S]
Step 1 runtime input0: [1, 4096]
  -> lookup/bind S = 4096, pass

Step 1 runtime input0: [1, 8192]
  -> bind S = 8192 conflicts with existing S = 4096, fail
```

当前 `ExecutionPlanBuilder` 接收扁平 `std::vector<ExecutionPlanNodeSpec>`，缺少图边信息，不能可靠判断 step 之间哪个 output 连接到哪个 input。因此 Phase 1 不做跨 step symbol unification。

如果暂不实现跨 step symbol unification，`ShapeBindingContext` 会退化为 per-step 验证语义：每个 step 只验证自己的 `input_specs` / `output_specs` 与当前 runtime tensor 是否一致；跨 step 一致性需要通过显式 `runtime_checks` 或 tensor 连接层的 shape 传递机制保障。

Phase 2 需要引入图级 shape environment：

```text
GraphEdge(output_node, output_idx) -> InputPort(input_node, input_idx)
```

只有明确图边证明两个 tensor dimension 来源相同，才能写入全局 solver 或复用相同 `ShapeSymbol`。禁止通过“相邻 step”推断跨 step equality。

## 十、Operator 改造示例：RmsNormOp

RmsNorm 的 hidden-size equality 是 requirement，不是 proven fact。

```cpp
StatusOr<InferenceResult> RmsNormOp::InferOutputShapes(
        std::span<const TensorSpec> inputs,
        ShapeInferenceContext& context) const {
    UNUSED(context);

    const auto& input = inputs[0];
    const auto& weight = inputs[1];

    const ShapeSymbol seq_dim = input.shape[0];
    const ShapeSymbol hidden_dim = input.shape[1];

    SymbolicShape output_shape({seq_dim, hidden_dim});

    ShapeConstraint hidden_constraint{
            .condition = DimEqualConstraint{
                    .lhs = DimLocator{TensorPort{TensorPortType::kInput, 0}, 1},
                    .rhs = DimLocator{TensorPort{TensorPortType::kInput, 1}, 0},
            },
            .error_context = "RmsNorm hidden_size mismatch",
    };

    return InferenceResult{
            .outputs = {TensorSpec{input.dtype, output_shape}},
            .runtime_checks = {std::move(hidden_constraint)},
    };
}
```

`ExecutionPlanBuilder` 随后调用 `ProveShapeConstraints`：

- 如果 input hidden 与 weight len 是同一个 symbol 或已被 solver 证明相等：删除 check。
- 如果是不同 symbolic 且无法证明：保留 runtime check。
- 如果 static mismatch：plan-time fail。

## 十一、dense vector solver 与生命周期风险

当前 `SymbolConstraintSolver` 使用 dense vector，并通过全局递减 `ShapeSymbol` value 映射索引：

```text
-2 → 0
-3 → 1
-4 → 2
```

因此不推荐“每个 Operator 创建独立 solver”。如果全局已经创建了大量 symbol，后面的 per-op solver 只要接触一个较新的 symbol，就可能扩容到很大的索引。

Phase 1C 推荐由 graph compiler / upgraded `ExecutionPlanBuilder` 持有长生命周期 solver。若未来仍需要短生命周期 solver，应先改造 solver 存储：

- 改回 sparse map；或
- 增加 local remap：global symbol value → compact local index。

## 十二、非 shape 结构性保障

以下内容不属于 shape proof/runtime validation 的第一闭环，但属于执行计划安全性的后续结构性保障。

### 12.1 Packed weights 生命周期绑定

`ExecutionStep.packed_weights` 当前是裸指针，依赖外部 `ModelInstance` 或 mmap backing 生命周期。后续应改为结构性所有权保障，例如：

```cpp
struct ExecutionStep {
    std::shared_ptr<const void> packed_weights_holder;
    const void* packed_weights = nullptr;
};
```

如果 packed weights 来自 mmap，则 `ExecutionPlan` 可以持有 backing 对象的 `shared_ptr`，热路径继续使用裸指针快速访问。

### 12.2 Workspace 边界防护

当前 `RuntimeBindingContext::BindWorkspace()` 已经检查 workspace arena 是否能满足 `WorkspaceRequirement`。后续可以增强 `WorkspaceBinding`：

```cpp
struct WorkspaceBinding {
    void* data = nullptr;
    size_t size = 0;
    size_t offset = 0;
    size_t arena_total = 0;

    AM_NODISCARD bool WithinArena() const noexcept {
        return offset <= arena_total && size <= arena_total - offset;
    }
};
```

Debug / Paranoid 模式下可以再考虑 guard page，但不作为当前阶段默认目标。

### 12.3 Kernel 密封

`Prepare()` 阶段已经把 resolved kernel 缓存在 operator 内部，普通执行不再查询 registry。因此 kernel registry 后续变化通常不会影响已构建 plan。

如需进一步防御 operator 内部状态被破坏，可以记录：

```cpp
struct KernelSeal {
    KernelFunc fn = nullptr;
    const char* debug_name = nullptr;
    size_t attrs_hash = 0;
};
```

该方案优先级低于 TensorSpec runtime validation、weights 生命周期和 workspace 边界。

## 十三、实施路线

### Phase 1A：Runtime TensorSpec validation 闭环

1. `ExecutionStep` 增加 `input_specs`，并继续保存 `output_specs` / `runtime_checks`。
2. `ExecutionPlanBuilder` 从 `ExecutionPlanNodeSpec.input_specs` 保存 step input specs。
3. 将 `ValidateShapeConstraints` 重命名为 `ValidateRuntimeShapeConstraints`，明确它只处理 concrete runtime checks。
4. 新增 TensorSpec runtime validator：
   - 验证 dtype。
   - 验证 rank。
   - 验证 static dim。
   - 用 per-Execute `ShapeBindingContext` 验证 generated symbol identity。
   - 忽略 `Unknown(-1)` 的 identity 绑定。
5. `LayerRunner::Run()` 创建 per-Execute `ShapeBindingContext`。
6. `LayerRunner::RunStep()` 在 `ValidateRuntimeShapeConstraints(runtime_checks)` 之前调用 TensorSpec validator。
7. 改造 runtime validation 相关 tests。

此阶段不引入 `ModelGraph`，不要求 `ShapeInferenceContext`，不做 solver proof 裁剪。

### Phase 1B：最小 ModelGraph / ValueId / ShapeEnvironment

1. 新增最小 `ModelGraph` 模块：
   - `NodeId`。
   - `ValueId`。
   - node inputs / outputs。
   - graph inputs / outputs。
   - producer-consumer edges。
   - value -> `TensorSpec` / shape facts。
   - topological order。
2. 新增 graph lowering 到 `ExecutionPlan` 的路径。
3. 让相同 graph value 的同一维度复用同一个 `ShapeSymbol::value()`，或由 `ShapeEnvironment` 显式记录 proven facts。
4. 保留当前 flat `ExecutionPlanNodeSpec` builder 作为低级/测试入口。

### Phase 1C：SymbolConstraintSolver plan-time proof

1. 新增 `ShapeInferenceContext`，由 graph compiler / upgraded `ExecutionPlanBuilder` 持有。
2. 修改 `Operator::InferOutputShapes` 签名并适配调用点。
3. 重构 `shape_constraint_evaluator` API：
   - 删除 public symbolic/runtime `EvaluateShapeConstraint` overload。
   - 新增 `ProveShapeConstraints`。
4. 基于 graph value identity / ShapeEnvironment 向 solver 写入 proven facts。
5. 调用 `ProveShapeConstraints` 裁剪 runtime checks。
6. 改造 operators：
   - `CheckInputSpecs` 收窄到非 solver 检查。
   - `InferOutputShapes` 返回待验证 constraints。
7. 改造 solver-aware proof tests。

### Phase 2：结构性执行安全

1. packed weights 生命周期绑定。
2. workspace boundary 信息增强。
3. kernel seal。
4. 必要时引入 `VerificationLevel`。

## 十四、测试计划

### 14.1 TensorSpec runtime validation 测试（Phase 1A）

1. dtype mismatch 失败。
2. rank mismatch 失败。
3. static dim mismatch 失败。
4. 同一 generated symbol 绑定到相同 concrete value 成功。
5. 同一 generated symbol 绑定到不同 concrete value 失败。
6. `Unknown(-1)` 不传播 identity。

### 14.2 Runtime shape constraint validation 测试（Phase 1A）

1. `ValidateRuntimeShapeConstraints` 满足约束时返回 Ok。
2. concrete dim mismatch 返回 `InvalidArgument`。
3. runtime validation 不返回、不暴露 `kDeferred`。

### 14.3 ModelGraph / ValueId 测试（Phase 1B）

1. graph input / output value 具有稳定 `ValueId`。
2. node input 可以追溯到 producer output。
3. topo order 能覆盖 producer-before-consumer。
4. 同一 `ValueId` 的 TensorSpec 复用同一 shape fact。
5. flat `ExecutionPlanNodeSpec` builder 低级入口保持可用。

### 14.4 Solver-aware proof 测试（Phase 1C）

1. `DimEqual`：同一 symbol → proof satisfied，runtime check 被删除。
2. `DimEqual`：不同 symbolic 无 proven fact → proof deferred，runtime check 保留。
3. `DimEqual`：symbol 绑定 static 8，与 static 8 → satisfied。
4. `DimEqual`：symbol 绑定 static 8，与 static 16 → violated。
5. `DimBroadcastable`：任一侧 proven static 1 → satisfied。
6. `DimBroadcastable`：proven static 2 vs 3 → violated。
7. `VolumeEqual`：逐位 solver-proven equal → satisfied。
8. `VolumeEqual`：两侧均可用 static binding 计算 volume → satisfied / violated。
9. `VolumeEqual`：存在未绑定 symbolic → deferred。

### 14.5 Operator / plan 回归测试（Phase 1C）

1. RmsNorm 两个不同动态符号：必须保留 runtime check。
2. RmsNorm 同一 hidden symbol：runtime check 可删除。
3. RmsNorm static hidden mismatch：plan-time fail。
4. Embedding 不需要额外 runtime check 的路径保持不变。
5. `ExecutionStep` 保存 input/output specs。
6. `LayerRunner` 在 runtime checks 前执行 TensorSpec validation。

## 十六、核心原则

1. Solver 是 plan-time proof 工具，不是 runtime binder。
2. Solver 只保存 proven facts，不保存普通 operator requirements。
3. Requirement 通过 `ShapeConstraint` 表达；无法 plan-time 证明时保留 runtime check。
4. `kSatisfied` 不进入 `runtime_checks`，但其证明前提必须由 TensorSpec runtime validation 保障。
5. Runtime validation 只处理 concrete TensorView，不接触 solver。
6. `ExecutionStep` 保存 input/output specs 和 runtime checks，不保存 solver。
7. `InferenceResult` 不持有 solver。
8. `ShapeBindingContext` 是 per-Execute 临时状态，不是 graph solver。
9. Phase 1A 先做 runtime TensorSpec validation，不依赖 ModelGraph。
10. Solver proof 应在最小 ModelGraph / ValueId / ShapeEnvironment 建立后实施。
11. 跨 step equality 必须基于真实图边实现，禁止通过 step 顺序推断。
12. 不保留 ambiguous `EvaluateShapeConstraint` symbolic/runtime public overload。
