# ADR-004: MADV_DONTNEED Strategy for Physical Memory Release with VMA Retention

> 用途：记录 `ammalloc` 已确认的重要设计决策。

- 标题：`MADV_DONTNEED Strategy for Physical Memory Release with VMA Retention`
- 状态：`accepted`
- 作用范围：
  - 影响模块：`ammalloc`
  - 影响子模块：`page_cache`, `page_heap_scavenger`
  - 关联记忆：
    - [`../module.md`](../module.md)
    - [`../submodules/page_cache.md`](../submodules/page_cache.md)
    - [`../submodules/page_heap_scavenger.md`](../submodules/page_heap_scavenger.md)
- 背景：
  - `munmap` 会完全移除虚拟内存映射；后续再次分配同一地址范围时，需要重新执行完整的 `mmap` 路径。
  - 工程说明明确指出，`mmap` / `munmap` 会竞争内核中较重的 `mmap_sem` 锁；频繁的映射与解除映射会形成内核侧瓶颈。
  - `MADV_DONTNEED` 可以释放物理页并降低 RSS，但保留原有虚拟内存区域 (VMA)。
  - 对已 `MADV_DONTNEED` 的地址再次访问时，只需要走缺页处理，而不是重新建立完整映射。
  - `ammalloc` 的页释放策略已经把 `madvise` 作为与操作系统交互的一部分，后台 `PageHeapScavenger` 也会对空闲且仍处于 committed 状态的页执行 `madvise(MADV_DONTNEED)`。
- 决策：
  - 对空闲或可回收的页缓存区域，优先调用 `madvise(MADV_DONTNEED)` 释放物理内存，而不是立即 `munmap`。
  - 释放后保留原有 VMA；虚拟地址区间继续存在，只回收底层物理页。
  - 后续若再次使用该地址范围，允许通过缺页恢复物理页，而不是重新走完整 `mmap` 分配。
  - 页释放策略以减少 `mmap_sem` 竞争为目标；后台清理线程可以周期性对满足条件的空闲 `Span` 执行该操作。
  - 该策略用于物理内存回收与页缓存复用场景；真正需要永久归还映射时，仍保留 `munmap` 路径。
- 原因：
  - 锁规避：工程说明已确认，保留 VMA 后，后续重用地址时可绕过沉重的 `mmap_sem` 竞争路径。
  - 性能：再次访问只需处理缺页，而不是重新执行完整的 `mmap` 操作。
  - VMA 稳定性：避免频繁 `munmap` / `mmap` 带来的映射拆建与额外内核开销。
  - RSS 控制：`MADV_DONTNEED` 仍可把物理页还给操作系统，降低进程驻留集大小。
  - 清理线程契合度：`PageHeapScavenger` 已被设计为周期性扫描空闲 `Span` 并在锁外执行 `madvise(MADV_DONTNEED)`。
- 影响：
  - 正向影响：减少 `mmap_sem` 相关争用，有利于多线程下的页回收与再分配路径。
  - 正向影响：最近释放区域的再次使用可复用现有 VMA，仅承担缺页代价。
  - 正向影响：降低内核侧映射拆建开销，贴合 `PageCache` 与后台 scavenger 的回收模型。
  - 代价：虚拟地址空间不会因释放物理页而立即消失；该取舍依赖 64 位地址空间充足这一前提。
  - 代价：`MADV_DONTNEED` 之后的首次访问会触发缺页开销。
  - 约束：该策略适用于期望后续仍可能复用的空闲页；并不替代永久释放映射的场景。
  - 约束：超大块或需要真正销毁映射的路径，仍需继续使用系统释放接口。
