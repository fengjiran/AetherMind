# ADR-005: No STL in Core Allocation Paths to Prevent Bootstrapping Deadlock

> 用途：记录 `ammalloc` 已确认的重要设计决策。

- 标题：`No STL in Core Allocation Paths to Prevent Bootstrapping Deadlock`
- 状态：`accepted`
- 作用范围：
  - 影响模块：`ammalloc`
  - 影响子模块：`thread_cache`, `central_cache`, `page_cache`
  - 关联记忆：
    - [`../module.md`](../module.md)
    - [`../submodules/thread_cache.md`](../submodules/thread_cache.md)
    - [`../submodules/central_cache.md`](../submodules/central_cache.md)
    - [`../submodules/page_cache.md`](../submodules/page_cache.md)
- 背景：
  - `ammalloc` 的核心分配/释放路径与元数据管理需要避免再次落回系统 `malloc`。
  - 若核心路径使用 `std::vector`、`std::string`、`std::map`、`std::unordered_map` 等 STL 堆分配容器，这些容器会触发系统 `malloc`。
  - 系统 `malloc` 已被 `am_malloc` Hook 拦截；一旦回到 `ammalloc`，核心路径又可能再次申请元数据。
  - 若这些元数据管理继续依赖 STL 容器或原生 `new` / `delete`，就会形成递归调用链。
  - 该递归会导致无限递归与栈溢出；工程约束将其定义为 Bootstrapping Problem / No `malloc` recursion。
- 决策：
  - 在核心分配/释放路径与元数据管理中，严格禁止使用会触发堆分配的 STL 容器，如 `std::vector`、`std::string`、`std::map`、`std::unordered_map` 等。
  - 在同一范围内，严格禁止使用原生 `new` / `delete` 操作符。
  - 核心路径统一采用自定义结构：`FreeList` 使用嵌入式链表，`TransferCache` 使用固定容量指针数组，`PageMap` 使用基数树。
  - `Span`、`RadixNode` 等元数据统一通过自定义 `ObjectPool` 管理。
  - 不需要动态增长时优先使用定长栈数组。
  - 不处于核心拦截路径的初始化、测试和 benchmark 不在本禁令覆盖范围内。
- 原因：
  - 破坏递归链：禁止 STL 堆分配容器与 `new` / `delete` 后，核心路径不会因为再次触发系统 `malloc` 而陷入递归。
  - 保持自包含：设计目标明确要求核心分配逻辑不依赖 STL 堆分配容器，以满足零外部依赖约束。
  - 分配行为可预测：定长数组、嵌入式链表和对象池的内存来源与生命周期都由 `ammalloc` 自身控制。
  - 贴合性能目标：`FreeList`、`TransferCache` 和 `PageMap` 都围绕固定布局与 O(1) 快路径设计。
  - 降低重入风险：该约束直接消除一类由分配器重入引起的崩溃问题。
- 影响：
  - 正向影响：从架构上切断 Bootstrapping Problem 导致的无限递归与栈溢出风险。
  - 正向影响：核心路径统一复用 `FreeList`、`TransferCache`、`PageMap` 与 `ObjectPool` 等自定义结构，内存布局与分配模式更可控。
  - 正向影响：延续零外部依赖与热路径 O(1) 的设计方向。
  - 代价：需要持续维护 `FreeList`、`SpanList`、`TransferCache`、`PageMap` 与 `ObjectPool` 等专用数据结构。
  - 代价：相比直接复用 STL，核心路径实现与评审都需要额外关注所有权、生命周期与分配来源。
  - 约束：所有涉及 `thread_cache`、`central_cache`、`page_cache` 的核心路径改动都必须检查是否引入 STL 堆分配容器或 `new` / `delete`。
  - 约束：新贡献者需要先理解该自举约束，再修改核心分配与元数据代码。
