# ADR-002: 4-Level RadixTree PageMap for Sparse 64-bit Address Space

> 用途：记录 `ammalloc` 已确认的重要设计决策。

- 标题：`4-Level RadixTree PageMap for Sparse 64-bit Address Space`
- 状态：`accepted`
- 作用范围：
  - 影响模块：`ammalloc`
  - 影响子模块：`page_cache`, `page_allocator`
  - 关联记忆：
    - [`../module.md`](../module.md)
    - [`../submodules/page_cache.md`](../submodules/page_cache.md)
- 背景：
  - 64 位系统在启用 ASLR 时会产生高位且极度稀疏的虚拟地址映射。
  - `PageMap` 需要承担 `PageID -> Span*` 映射，用于释放路径中的归属查询。
  - 若为 `PageID -> Span*` 采用巨大扁平数组，稀疏地址空间下会带来过高内存占用与 OOM 风险。
  - `am_free` 相关路径依赖 `GetSpan`，因此需要无锁读取；`SetSpan` 与 `ClearRange` 可以接受受保护的慢写路径，但必须保证正确性。
- 决策：
  - `PageMap` 采用 4 层基数树结构：`RadixRootNode` 胖根节点加 3 层 `RadixNode`。
  - 该结构覆盖 48-bit 虚拟地址空间；通过胖根节点设计可扩展覆盖 57-bit 地址空间。
  - 读取路径 `GetSpan` 采用无锁遍历。
  - 写入路径 `SetSpan`、`ClearRange` 统一受 `PageCache` 互斥锁保护。
  - 基数树节点按需增长，只在写入时分配新节点，不回收单个 `RadixNode`。
  - `PageMap` 相关元数据仅在系统关闭时通过 `ObjectPool::ReleaseMemory` 统一释放。
- 原因：
  - 内存效率：仅为实际使用到的地址范围分配树节点，避免扁平数组在稀疏地址空间下浪费大量内存。
  - ASLR 兼容性：能够处理 64 位系统中高位、稀疏的地址分布，而不需要预留巨大连续映射表。
  - 性能：`GetSpan` 保持无锁读取，贴合释放快路径对低延迟归属查询的需求。
  - 简化实现：树结构只增不减，避免删除节点带来的回收、同步与悬空指针复杂度。
  - 并发安全：沿用 `PageCache` 现有互斥锁串行化写操作，保持读写职责清晰。
- 影响：
  - 正向影响：`PageMap` 可以在 64 位稀疏地址空间下维持较低元数据占用，避免因扁平数组而造成过量内存消耗。
  - 正向影响：`GetSpan` 保持无锁读取，有利于 `am_free` 路径中的快速归属查询。
  - 代价：查询需要经过固定层级的树遍历，相比扁平数组索引会增加指针跳转。
  - 代价：树节点不会在运行期回收，相关元数据仅在系统关闭时统一释放。
  - 约束：所有 `PageMap` 写操作都必须在 `PageCache` 互斥锁保护下执行。
  - 约束：`RadixNode` 对象不得被单独显式释放。
