# ADR-003: Optimistic Huge Page Strategy for Reduced VMA Fragmentation

> 用途：记录 `ammalloc` 已确认的重要设计决策。

- 标题：`Optimistic Huge Page Strategy for Reduced VMA Fragmentation`
- 状态：`accepted`
- 作用范围：
  - 影响模块：`ammalloc`
  - 影响子模块：`page_allocator`
  - 关联记忆：
    - [`../module.md`](../module.md)
    - [`../submodules/page_allocator.md`](../submodules/page_allocator.md)
- 背景：
  - 传统做法是为了获得 `2MB` 对齐的大页地址而先多申请再裁剪。
  - `ammalloc` 的工程说明明确指出，这种 `over-allocate & trim` 会带来 VMA (Virtual Memory Area) 碎片和额外系统调用开销。
  - 额外的裁剪 `munmap` 还会增加内核 `mmap_sem` 锁压力。
  - `PageAllocator` 设计已包含 Huge Page 支持；`2MB` 大页用于降低 TLB 压力。
  - 在启用 THP (Transparent Huge Pages) 的系统上，直接申请后自然得到 `2MB` 对齐地址是常见情况。
- 决策：
  - `PageAllocator` 采用乐观大页策略，先按目标大小直接执行一次 `mmap` 申请。
  - 若返回地址天然满足 `2MB` 对齐，则直接保留并按大页路径使用该区域。
  - 不采用以强制对齐为目的的“多申请再裁剪”默认模式。
  - `HugePageCache` 继续管理和复用命中的 `2MB` 大页区域。
  - 若返回地址未对齐，则该映射直接按普通页使用，不强制追求大页对齐。
- 原因：
  - THP 兼容性：工程说明已确认，在启用 THP 的系统上，直接申请后命中 `2MB` 对齐地址很常见。
  - 降低碎片：不做 `over-allocate & trim`，可避免额外的 VMA 拆分与碎片化。
  - 降低系统调用开销：相比申请后再裁剪，可减少额外 `munmap`。
  - 降低内核锁压力：减少 `mmap_sem` 相关压力，有利于长期运行和并发场景。
  - 简化实现：路径只需检查对齐结果并选择大页或普通页处理，无需复杂裁剪逻辑。
- 影响：
  - 正向影响：长时间运行时可减少因对齐裁剪带来的 VMA 碎片累积。
  - 正向影响：大页命中时可结合 `HugePageCache` 复用 `2MB` 区域，并利用大页降低 TLB 压力。
  - 正向影响：减少额外 `munmap`，从而降低系统调用数量。
  - 代价：当 THP 未启用或 OS 未自然返回 `2MB` 对齐地址时，大页命中率会低于激进对齐方案。
  - 代价：大页可用性依赖 OS 返回地址，对齐结果不如强制裁剪方案确定。
  - 约束：最佳效果依赖 THP 与 OS 地址返回行为；运行期是否命中大页具有非确定性。
