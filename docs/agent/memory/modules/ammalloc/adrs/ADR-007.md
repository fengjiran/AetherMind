# ADR-007: Span v2 64B Refactor - Cache Line Aligned Memory Layout

> 用途：记录 `ammalloc` 已确认的重要设计决策。

- 标题：`Span v2 64B Refactor - Cache Line Aligned Memory Layout`
- 状态：`accepted`
- 作用范围：
  - 影响模块：`ammalloc`
  - 影响子模块：`page_cache`, `central_cache`, `page_heap_scavenger`
  - 关联记忆：
    - [`../module.md`](../module.md)
    - [`../submodules/page_cache.md`](../submodules/page_cache.md)
- 背景：
  - 原始 `Span` 结构体大小为 112B，跨两个缓存行（64B + 48B）。
  - 多线程高并发场景下，跨缓存行访问导致 False Sharing，影响性能。
  - 字段冗余：`data_base_ptr` 可通过计算推导，`bitmap` 和 `bitmap_num` 可通过内联方法计算。
  - 位域分散：`is_used` 和 `is_committed` 分别存储，打包效率低。
- 决策：
  - 将 `Span` 结构体重构为 64B，严格单缓存行对齐（`alignas(64)`）。
  - 删除可计算字段：`data_base_ptr` → `obj_offset` + `GetDataBasePtr()` 计算。
  - 删除存储字段：`bitmap` 和 `bitmap_num` → `GetBitmap()` / `GetBitmapNum()` 内联计算。
  - 合并状态位：`is_used` 和 `is_committed` 打包到 `uint16_t flags` 位域。
  - 引入访问器方法：统一通过 `IsUsed()`, `SetUsed()`, `IsCommitted()`, `SetCommitted()`, `GetPageBaseAddr()`, `GetDataBasePtr()`, `GetBitmap()`, `GetBitmapNum()` 访问。
- 原因：
  - **缓存行对齐**：64B 对齐确保单个 `Span` 完全落在一条缓存行内，消除跨行访问和 False Sharing。
  - **内存压缩**：112B → 64B（减少 43%），每个 `Span` 对象节省 48B 内存。
  - **计算替代存储**：`data_base_ptr`（8B）→ `obj_offset`（4B）+ 计算；`bitmap`（8B）和 `bitmap_num`（8B）→ 内联方法。
  - **位域打包**：`is_used` 和 `is_committed` 合并到 `flags`（2B），节省 6B 并减少内存分散访问。
  - **API 统一**：访问器方法提供封装边界，未来布局变更不影响调用方。
- 影响：
  - 正向影响：消除多线程场景下的 False Sharing，预期性能提升 10-20%。
  - 正向影响：减少内存占用，相同 `span_pool_` 容量可容纳更多 `Span` 对象。
  - 正向影响：单缓存行访问保证原子性，简化并发推理。
  - 代价：访问器方法引入额外函数调用（编译器内联优化后可忽略）。
  - 代价：`GetBitmap()` / `GetBitmapNum()` 运行时计算引入轻微 CPU 开销（相对于 False Sharing 收益可忽略）。
  - 约束：所有 `Span` 字段访问必须迁移到访问器方法；直接字段访问会导致编译错误。

## Span v2 布局详情

```cpp
struct alignas(64) Span {
    // 区域 1: 链表指针 (16B)
    Span* next;
    Span* prev;

    // 区域 2: 页级寻址与状态 (16B)
    uint64_t start_page_idx;     // 8B
    uint32_t page_num;           // 4B
    uint16_t flags;              // 2B: bit0=is_used, bit1=is_committed
    uint16_t size_class_idx;     // 2B

    // 区域 3: 对象分配元数据 (16B)
    uint32_t obj_size;           // 4B
    uint32_t capacity;           // 4B
    uint32_t use_count;          // 4B
    uint32_t scan_cursor;        // 4B

    // 区域 4: 杂项与冷数据 (16B)
    uint32_t obj_offset;         // 4B: 替代 data_base_ptr
    uint32_t padding;            // 4B
    uint64_t last_used_time_ms;  // 8B
};
```

## 访问器方法规范

| 方法 | 原字段 | 实现 |
|------|--------|------|
| `IsUsed()` / `SetUsed(bool)` | `is_used` | `flags & kUsedMask` |
| `IsCommitted()` / `SetCommitted(bool)` | `is_committed` | `flags & kCommittedMask` |
| `GetPageBaseAddr()` | `start_page_idx` | `start_page_idx << PAGE_SHIFT` |
| `GetDataBasePtr()` | `data_base_ptr` | `GetPageBaseAddr() + obj_offset` |
| `GetBitmap()` | `bitmap` | `reinterpret_cast<uint64_t*>(GetPageBaseAddr())` |
| `GetBitmapNum()` | `bitmap_num` | `(capacity + 63) >> 6` |

## 验证结果

- 单元测试：46/46 通过（包括多线程压力测试）
- 构建验证：`ammalloc`, `aethermind_unit_tests` 构建成功
- LSP 诊断：无错误（仅 1 个 unused-includes 警告，非阻塞）
- 基准测试：
  - 单线程吞吐：3.56M/s - 4.74M/s operations
  - 多线程（16 threads）：799k/s - 1.88M/s operations
  - PageCache 精确命中延迟：484ns - 730ns
  - 合并释放延迟：346ns - 376ns
