# 全面代码审查指南

> 适用于 AetherMind 项目及其他 C++ 项目的系统化代码审查方法

---

## 审查维度框架

```
┌─────────────────────────────────────────────────────────────┐
│                    全面代码审查 checklist                      │
├─────────────────────────────────────────────────────────────┤
│ 1. 功能性 (Functionality)        - 是否做了正确的事           │
│ 2. 设计/架构 (Design/Architecture) - 是否以正确的方式组织      │
│ 3. 性能 (Performance)            - 是否足够快                 │
│ 4. 代码质量 (Code Quality)       - 是否整洁、可读             │
│ 5. 可维护性 (Maintainability)    - 是否易于修改               │
│ 6. 安全性 (Security)             - 是否安全                   │
│ 7. 测试 (Testing)                - 是否被充分验证             │
│ 8. 文档 (Documentation)          - 是否被充分记录             │
└─────────────────────────────────────────────────────────────┘
```

---

## 1. 功能性 (Functionality)

### 核心问题：**代码是否正确地实现了需求？**

| 检查点 | 具体问题 | 审查方法 |
|--------|----------|----------|
| **正确性** | 算法逻辑是否正确？ | 代码走读、数学推导、边界测试 |
| **完整性** | 是否处理了所有边界情况？ | 检查边界条件、异常路径 |
| **一致性** | 行为是否与文档/接口承诺一致？ | 对比接口文档 |
| **鲁棒性** | 异常输入如何处理？ | 检查错误处理、返回值 |
| **副作用** | 是否有意外的副作用？ | 检查全局状态修改 |

### 示例检查清单

- [ ] 空指针/空值检查
- [ ] 数组/容器越界检查
- [ ] 整数溢出检查
- [ ] 资源泄漏检查
- [ ] 并发安全性检查

---

## 2. 设计/架构 (Design/Architecture)

### 核心问题：**代码结构是否合理、可扩展？**

| 检查点 | 具体问题 | 审查方法 |
|--------|----------|----------|
| **单一职责** | 类/函数是否只做一件事？ | 检查类大小、函数长度 |
| **依赖关系** | 依赖是否合理？有无循环依赖？ | 绘制依赖图 |
| **抽象层次** | 接口与实现分离是否清晰？ | 检查 public/private |
| **扩展性** | 新增功能是否需要大量修改？ | 开闭原则检查 |
| **内聚性** | 相关功能是否组织在一起？ | 检查命名空间、模块 |

### SOLID 原则检查

- [ ] **S**ingle Responsibility: 单一职责
- [ ] **O**pen/Closed: 对扩展开放，对修改关闭
- [ ] **L**iskov Substitution: 里氏替换
- [ ] **I**nterface Segregation: 接口隔离
- [ ] **D**ependency Inversion: 依赖倒置

### 其他原则

- [ ] DRY (Don't Repeat Yourself)
- [ ] KISS (Keep It Simple, Stupid)

---

## 3. 性能 (Performance)

### 核心问题：**代码是否足够高效？**

| 检查点 | 具体问题 | 审查方法 |
|--------|----------|----------|
| **时间复杂度** | 算法复杂度是否合理？ | 大 O 分析 |
| **空间复杂度** | 内存使用是否合理？ | 检查数据结构大小 |
| **热点优化** | 热路径是否有优化？ | Profiling 数据 |
| **内存布局** | 缓存友好性如何？ | 检查数据对齐、局部性 |
| **资源管理** | 资源获取/释放是否高效？ | RAII、池化技术 |

### 性能检查清单

- [ ] 避免不必要的拷贝（移动语义）
- [ ] 减少动态内存分配
- [ ] 减少锁竞争
- [ ] 减少系统调用
- [ ] 利用编译期计算（constexpr/consteval）

---

## 4. 代码质量/风格 (Code Quality/Style)

### 核心问题：**代码是否整洁、可读、一致？**

| 检查点 | 具体问题 | 审查方法 |
|--------|----------|----------|
| **命名** | 变量/函数/类名是否清晰？ | 自解释命名检查 |
| **格式** | 是否符合项目规范？ | 对比 `.clang-format` |
| **长度** | 函数/类是否过大？ | 函数 < 50 行，类 < 300 行 |
| **嵌套** | 嵌套层次是否过深？ | 避免 > 3 层嵌套 |
| **注释** | 注释是否清晰、必要？ | 注释比例、质量 |

### C++ 特定检查

- [ ] 使用现代 C++ 特性（C++11/14/17/20/23）
- [ ] 避免裸指针（使用智能指针）
- [ ] 正确使用 const/constexpr/noexcept
- [ ] 避免隐式类型转换（explicit）
- [ ] 遵循 Rule of 0/3/5

---

## 5. 可维护性 (Maintainability)

### 核心问题：**代码是否易于理解和修改？**

| 检查点 | 具体问题 | 审查方法 |
|--------|----------|----------|
| **可读性** | 代码是否自解释？ | 让同事阅读测试 |
| **模块化** | 修改是否影响最小范围？ | 检查耦合度 |
| **可测试性** | 是否易于单元测试？ | 检查依赖注入 |
| **可调试性** | 出现问题是否易于调试？ | 日志、断言检查 |
| **版本兼容** | 修改是否破坏兼容性？ | API 稳定性检查 |

---

## 6. 安全性 (Security)

### 核心问题：**代码是否存在安全漏洞？**

| 检查点 | 具体问题 | 审查方法 |
|--------|----------|----------|
| **输入验证** | 是否信任外部输入？ | 检查所有输入点 |
| **缓冲区溢出** | 数组访问是否安全？ | 边界检查 |
| **并发安全** | 是否存在数据竞争？ | 静态分析、代码审查 |
| **资源泄漏** | 是否存在泄漏？ | 内存/文件/锁检查 |
| **异常安全** | 异常时资源是否正确释放？ | 强异常安全保证 |

### C++ 安全最佳实践

- [ ] 使用 RAII 管理资源
- [ ] 使用标准库容器代替裸数组
- [ ] 避免使用 `new`/`delete`，使用智能指针
- [ ] 检查所有 `reinterpret_cast`
- [ ] 使用 `std::string_view` 代替裸指针

---

## 7. 测试 (Testing)

### 核心问题：**代码是否被充分验证？**

| 检查点 | 具体问题 | 审查方法 |
|--------|----------|----------|
| **覆盖率** | 单元测试覆盖了多少？ | 覆盖率报告 |
| **边界测试** | 边界条件是否被测试？ | 检查测试用例 |
| **异常测试** | 错误路径是否被测试？ | 负面测试 |
| **并发测试** | 并发场景是否被测试？ | TSan/压力测试 |
| **性能测试** | 性能回归是否被监控？ | Benchmark |

### 测试金字塔

```
      /\
     /  \     E2E 测试 (少数)
    /----\
   /      \   集成测试 (一些)
  /--------\ 
 /          \ 单元测试 (大量)
/------------\
```

## 8. 文档 (Documentation)

### 核心问题：**代码是否被充分记录？**

| 检查点 | 具体问题 | 审查方法 |
|--------|----------|----------|
| **API 文档** | 公共接口是否有文档？ | Doxygen/注释 |
| **设计文档** | 架构决策是否被记录？ | 设计文档 |
| **使用示例** | 是否有使用示例？ | README/示例代码 |
| **变更记录** | 变更是否被记录？ | CHANGELOG |
| **内部注释** | 复杂逻辑是否有注释？ | 代码审查 |

### 文档检查清单

- [ ] 所有公共函数有文档注释
- [ ] 复杂算法有解释性注释
- [ ] 非直观代码有注释说明"为什么"
- [ ] TODO/FIXME 有跟踪信息
- [ ] 外部依赖有说明

## 实用审查工具

### 静态分析

```bash
# C++ 静态分析
clang-tidy -checks='cppcoreguidelines-*,modernize-*,performance-*,portability-*,readability-*' src/*.cpp

# 格式化检查
clang-format -n --Werror $(git ls-files '*.h' '*.cpp')

# 内存检查 (Valgrind)
valgrind --leak-check=full ./test_binary

# 线程安全检查 (TSan)
./build-tsan/tests/unit/test_binary
```

### 动态分析

```bash
# 地址消毒 (ASan)
./build-asan/tests/unit/test_binary

# 未定义行为检测 (UBSan)
./build-ubsan/tests/unit/test_binary
```

## 审查流程建议

### Phase 1: 自动化检查 (5分钟)

- 编译检查
- 静态分析
- 单元测试

### Phase 2: 快速浏览 (10分钟)

- 理解整体结构
- 检查命名规范
- 发现明显问题

### Phase 3: 深入审查 (30分钟+)

- 逐行审查核心逻辑
- 检查边界情况
- 验证算法正确性
- 评估设计决策

### Phase 4: 反馈与迭代

- 整理审查意见
- 与作者讨论
- 验证修复

## AetherMind 项目特定检查

### ammalloc 子系统特殊约束

- [ ] **无递归 malloc**: 不使用 STL 堆容器、new/delete
- [ ] **缓存行对齐**: 高频结构 `alignas(64)`
- [ ] **显式内存序**: `std::atomic` 必须指定 memory_order
- [ ] **RadixTree 只增不减**: 禁止单个节点 delete

### 命名规范

- [ ] 类型: `PascalCase`
- [ ] 函数: `snake_case` (核心 API)
- [ ] 枚举: `kPrefix`
- [ ] 宏: `AM_` 前缀 + 大写下划线
- [ ] 头文件保护: `AETHERMIND_<NAME>_H`

### 错误处理

- [ ] 抛异常: `AM_THROW(ErrorKind) << message`
- [ ] 断言: `AM_CHECK`, `AM_DCHECK`
- [ ] 分支预测: `AM_LIKELY`, `AM_UNLIKELY`
