## 总体路线

建议分成 8 个阶段：

1. **M0：骨架与契约冻结**
2. **M1：模型加载链路打通**
3. **M2：最小 Reference Prefill/Decode 跑通**
4. **M3：Executor / KV / Workspace 收敛**
5. **M4：Reference 路径端到端稳定**
6. **M5：第一条 Optimized 路径落地**
7. **M6：API / C ABI / 工程化补齐**
8. **M7：Phase 1 收口验收**

这 8 个阶段里，真正不能跳过的主线只有三条：

- **加载链**：`config -> safetensors -> LoadedModel`
- **执行链**：`PrepareSession -> Prefill -> Decode`
- **验证链**：`operator correctness -> e2e regression -> zero-allocation`

------

## M0：骨架与契约冻结

### 目标

先把工程骨架、对象边界、头文件 contract 冻住，避免后面一边写一边改大接口。

### 要做什么

先建出目标目录骨架，至少覆盖 `runtime / model / execution / ops / dispatch / platform / c_api`，与架构文档保持一致。

优先冻结这些头文件接口：

- `runtime/runtime.h`
- `runtime/loaded_model.h`
- `runtime/session.h`
- `runtime/generation_config.h`
- `execution/executor.h`
- `execution/prefill_path.h`
- `execution/decode_path.h`
- `execution/kv_cache.h`
- `execution/kv_cache_view.h`
- `dispatch/dispatch_table.h`
- `platform/threading.h`

同时把这几类 contract 明确成代码接口，而不是只停留在文档里：

- `ExecutorContext`
- `SessionState`
- `SessionExecutionState`
- `DecodeStepOutput`
- `KVCacheView`
- `WorkspaceBinding`
- `Tensor/Buffer/View`

这一步的依据就是文档里已经冻结的实现级 contract 与执行器对象模型。

### 交付物

- 可编译的空工程
- 所有核心头文件
- 错误码/Status 基础设施
- 一份最小 build 脚本
- 一份 smoke test（空模型 / mock dispatch）

### 验收标准

- 工程能完整编译
- 不存在循环依赖式 include
- 头文件职责边界清楚
- 后续模块可以并行开发

------

## M1：模型加载链路打通

### 目标

把“模型文件”变成 “LoadedModel”。

### 要做什么

实现最小模型加载路径：

1. `config.json` 解析
2. `safetensors` header/metadata 解析
3. tensor manifest 建立
4. 原始权重视图加载
5. 重排/对齐到内部布局
6. 构造 `LoadedModel`

Phase 1 的对象模型已经要求至少区分 `Model Assets` 和 `Loaded Model`，并让 Runtime / ModelManager 管理生命周期。

这里不要一开始就做“通用 HF 全兼容”。
只支持当前冻结的 **Llama-family dense** 配置子集即可。

### 交付物

- `ConfigParser`
- `SafetensorsLoader`
- `WeightManifest`
- `RepackStore`
- `LoadedModel`

### 验收标准

- 能成功加载目标模型目录
- 能打印关键模型元数据
- 能校验不支持的结构并明确失败
- 所有张量都有稳定的内部命名和视图

### 风险点

- tensor 名映射混乱
- 量化权重 metadata 过早耦合到执行层

------

## M2：最小 Reference Prefill/Decode 跑通

### 目标

先用最朴素但正确的路径把 token-to-token 跑通。

### 要做什么

优先实现最小 Reference 算子集：

- Embedding
- RMSNorm
- RoPE
- Attention
- Linear
- SwiGLU
- Argmax

这是整体架构文档明确列出的最小算子集。

这一阶段不追求快，只追求：

- 语义正确
- shape/stride contract 正确
- Prefill 与 Decode 结果可回归

同时把 `Executor` 先用最简单方式跑通：

- `Generate()`
- `PrepareSession()`
- `Prefill()`
- `DecodeStep()`

并按执行器文档要求使用统一控制流，而不是两套独立执行器。

### 交付物

- Reference operator kernels
- 最小 `Executor`
- `PrefillPath`
- `DecodePath`
- 单层 `LayerRunner`

### 验收标准

- 给定一组 token ids，可以稳定生成输出 token ids
- Prefill 不是“循环调用 DecodeStep 伪造”的
- EOS / max_new_tokens 生效
- 单线程下结果稳定

------

## M3：Executor / KV / Workspace 收敛

### 目标

把执行层真正从“能跑”升级成“结构正确”。

### 要做什么

按执行器设计文档，把执行层收敛成：

- 一个 `Executor`
- 两条路径：`PrefillPath / DecodePath`
- 一套共享 `LayerRunner`
- 一套统一 `DispatchTable`

并补齐状态机、执行上下文和 session 侧绑定。

同时把 KV 子系统收敛为：

- `KVCacheManager`
- `KVLayoutContract`
- `KVCacheView`

并确保：

- Session 创建时一次性预分配
- K/V 独立连续区域
- 逻辑索引至少包含 `layer_idx / kv_head_idx / seq_pos / head_dim_idx`
- `Reset` 不重新分配
- `Executor` 和 `Attention` 只通过 `KVCacheView` 访问 KV。

Workspace 也在这一阶段定型：

- hidden buffer
- logits buffer
- Q/K/V scratch
- attention scratch

且全部在 `Session Preparation` 阶段绑定。

### 交付物

- `SessionExecutionState`
- `KVCacheManager`
- `KVCacheView`
- `WorkspaceBinding`
- `DispatchTable`

### 验收标准

- `PrepareSession()` 里完成 KV / Workspace 绑定
- Decode 稳态路径无新增堆分配
- Attention 历史读取和当前写入逻辑正确
- 旧 view 不会误用已 reset/release 的 KV

------

## M4：Reference 路径端到端稳定

### 目标

把“能跑通”变成“可验证、可回归、可定位问题”。

### 要做什么

按架构文档里测试 contract，把测试体系至少分成四层：

1. Operator Correctness
2. Reference vs Optimized Consistency（此时先占位）
3. End-to-End Generation Regression
4. Memory Behavior / zero-allocation

这是你文档已经冻结的测试要求。

除此之外，再补两类很实际的测试：

- Loader correctness：模型加载映射和维度检查
- KV correctness：Prefill/Decode/Reset/越界保护

KV 文档也明确要求布局正确性、越界保护、Reset 行为、零分配验证和与 Attention 的集成验证。

### 交付物

- 单元测试
- golden regression
- decode zero-allocation hook
- 失败时可读的日志 / 断言

### 验收标准

- 同输入重复运行结果稳定
- Golden case 可长期回归
- Decode 零分配测试能自动跑
- 关键 shape/stride 错误能快速暴露

------

## M5：第一条 Optimized 路径落地

### 目标

在不改控制流的前提下，落第一条性能路径。

### 要做什么

严格遵守：

- `reference first, optimized second`
- `init-time resolve, hot-path direct call`

也就是执行控制流不变，只通过 `DispatchTable` 替换算子实现。

我建议第一条优化路径这样选：

1. 先做 `Linear` 的 CPU feature-aware 优化
2. 再做 RMSNorm / RoPE / Argmax 的轻量优化
3. 最后再碰 Attention 优化

因为一上来优化 Attention，排错难度太高。

量化路径上，Phase 1 只要求“至少一条可验证的量化线性路径”，优先 INT8，可扩展到 INT4。
所以这里推荐：

- **先做 INT8 linear optimized path**
- INT4 作为 M5 后半段或 M6 补充

### 交付物

- CPU feature detect
- `DispatchKey`
- `KernelRegistry`
- `DispatchTable`
- 第一条 optimized linear
- 量化线性验证

### 验收标准

- 热路径不做动态查表
- Reference / Optimized 在容差内一致
- 性能有可观改善
- 没破坏端到端回归

------

## M6：API / C ABI / 工程化补齐

### 目标

让引擎具备“可集成、可调用、可测试”的基本产品形态。

### 要做什么

补齐两套边界：

#### C++ API

- Runtime
- LoadedModel
- Session
- GenerationConfig

#### Thin C ABI

- opaque handle
- create/load/generate/destroy/error

这也是架构文档明确列出的 Phase 1 必须具备项。

同时补充：

- benchmark driver
- CLI demo
- 基础 profiling hook
- 错误码与错误消息桥接

### 交付物

- `runtime.h`
- `loaded_model.h`
- `session.h`
- `am_runtime.h`
- CLI / benchmark 工具

### 验收标准

- C++ 可直接本地集成
- C ABI 可被简单外部程序调用
- benchmark 能跑出基础吞吐/延迟
- 错误处理可读

------

## M7：Phase 1 收口验收

### 目标

确认已经满足“Phase 1 完整交付”，而不是“功能散装可跑”。

### 必须逐条对照的验收项

#### 功能项

- `token-based runtime`
- `config + safetensors loader`
- `static KV cache`
- `reference kernels`
- `deterministic greedy generation`
- `C++ API + thin C ABI`
- 至少一条可验证量化线性路径。

#### 约束项

- 单进程、单模型、单请求、同步阻塞
- token ids in / token ids out
- Llama-family dense
- 不引入 scheduler / batching / paged attention / GPU / serving。

#### 执行项

- `Generate()` 主入口成立
- `Session Preparation -> Prefill -> DecodeStep...` 状态机稳定
- Decode 稳态零分配
- `KVCacheView` 是唯一 KV 访问边界。

#### 测试项

- Operator correctness
- End-to-end regression
- Zero-allocation
- Reference vs optimized consistency。

### 收口后的产出

- `v1.0 runtime core`
- 文档与代码一致
- 一组稳定 regression baseline
- 明确的 Phase 2 backlog

------

## 推荐的实际开发顺序

如果按最稳的实现顺序，我建议这样排：

1. **M0**：先把头文件和 contract 冻住
2. **M1**：打通 loader
3. **M2**：先跑通 reference e2e
4. **M3**：把 Executor/KV/Workspace 收敛成正确结构
5. **M4**：把测试与零分配打牢
6. **M5**：做第一条 optimized linear
7. **M6**：补 API/C ABI/benchmark
8. **M7**：Phase 1 收口

其中最重要的分界线是：

- **M2 完成**：说明“功能上能跑”
- **M4 完成**：说明“架构上站稳”
- **M7 完成**：说明“Phase 1 真正交付”

------

## 每阶段最该防的坑

### 1. 过早做泛化

现在不要做：

- 通用图执行器
- 多模型抽象
- request scheduler
- paged KV
- tokenizer/serving 一体化

因为这些都明确不在 Phase 1 范围内。

### 2. 过早做重优化

Reference path 没站稳前，不要急着把时间花在 attention SIMD 极限优化上。

### 3. 控制流和实现路径耦合

不要分成两套 Executor：
控制流必须只有一套，Reference / Optimized 只通过 dispatch 分化。

### 4. 把 KV 物理布局泄漏到执行层

一旦 Executor/Attention 写死底层 offset 规则，Phase 2 换 Paged KV 会很难。

### 5. 忽视零分配验证

“以为没有分配” 和 “被自动化验证没有分配” 是两回事。M4 前就要把钩子准备好。

------

## 一句话定版

这条 roadmap 的核心不是“尽快堆功能”，而是：

**先用 M0-M4 建立一个 contract 稳定、reference 正确、KV/Executor 边界清晰、Decode 零分配可验证的 CPU 本地推理底座，再用 M5-M7 把第一条优化路径、API 边界和 Phase 1 交付形态补齐。**
