# ADR-001: Introducing TransferCache in CentralCache for O(1) Batch Transfer

> 用途：记录 `ammalloc` 已确认的重要设计决策。

- 标题：`Introducing TransferCache in CentralCache for O(1) Batch Transfer`
- 状态：`accepted`
- 作用范围：
  - 影响模块：`ammalloc`
  - 影响子模块：`central_cache`
  - 关联记忆：
    - [`../module.md`](../module.md)
    - [`../submodules/central_cache.md`](../submodules/central_cache.md)
- 背景：
  - `CentralCache` 采用两层 bucket 架构：快路径是 `TransferCache`，慢路径是 `SpanList`。
  - 仅依赖 `SpanList` 时，批量对象流转需要在 `std::mutex` 保护下扫描 `Span` 位图，高并发场景下过慢。
  - `ThreadCache` 与 `CentralCache` 之间存在按 `batch_num` 批量取放对象的固定模式，需要一个不必为每个对象都触发位图扫描的快路径。
- 决策：
  - 在每个 `CentralCache::Bucket` 中引入独立的 `TransferCache` 层，作为 `ThreadCache` 与 `CentralCache` 之间的批量对象快缓存。
  - `TransferCache` 使用固定容量的指针数组实现，容量按 `8 * batch_size` 规划。
  - `TransferCache` 由轻量级 `SpinLock`（TTAS）保护，只负责对象指针的批量读写。
  - 只有当 `TransferCache` 为空或已满时，请求才回退到 `SpanList` 的 `std::mutex + Bitmap` 慢路径。
- 原因：
  - 设计文档已明确：在 `CentralCache` 的互斥锁下扫描 Bitmap 对高并发批量操作来说太慢，因此需要把这类流转改成 `SpinLock` 保护下的 O(1) 指针数组拷贝。
  - `SpinLock` 文档明确面向 `CentralCache` 桶锁这类极短临界区；`TransferCache` 只做指针数组读写，避免在快路径触碰位图等复杂元数据。
  - `FetchRange()` 在慢路径中会额外预取一批对象回填 `TransferCache`，为后续同尺寸请求准备快路径，继续摊还互斥锁开销。
  - 两层结构把“批量搬运指针”和“切分 Span / 扫描位图 / 与 PageCache 交互”分离，保持职责清晰。
- 影响：
  - 正向影响：`CentralCache` 的批量取放优先走 `TransferCache` 指针数组操作，不必在每次请求时都进入 `SpanList` 位图扫描。
  - 正向影响：bucket 内形成 `TransferCache(SpinLock)` 与 `SpanList(std::mutex)` 的双锁域，快慢路径边界更清晰。
  - 代价：每个 bucket 需要额外持有 `8 * batch_size * sizeof(void*)` 的连续指针存储。
  - 代价：`SpinLock` 在竞争期间会忙等，但该锁只覆盖极短的数组读写临界区。
  - 约束：`TransferCache` 容量有限，突发流量仍可能回退到 `SpanList`；慢路径仍需处理位图扫描、预取与回退对象的兜底逻辑。
