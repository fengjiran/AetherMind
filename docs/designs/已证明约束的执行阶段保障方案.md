# 已证明约束的执行阶段保障方案

## 一、问题定义

当前系统的形状约束采用三态模型：

| 评估结果 | 含义 | 处理方式 |
| --- | --- | --- |
| `kSatisfied` | 符号推导阶段已证明 | 需要在执行阶段验证其前提没有漂移 |
| `kViolated` | 符号推导阶段已发现违规 | 构建失败 |
| `kDeferred` | 符号推导阶段无法确定 | `runtime_checks` 在执行时验证 |

核心问题是：`runtime_checks` 只覆盖 `kDeferred` 约束。对于 `kSatisfied` 约束，符号推导阶段实际依赖的是 `TensorSpec` 中的 dtype、rank、static dim 和 `ShapeSymbol` identity。如果同一个 `ExecutionPlan` 在执行阶段绑定了不满足这些前提的 tensor，已证明约束就可能被破坏。

典型例子：

```text
Plan-time:
  input0: [S, H]
  input1: [H]
  => input0.dim1 == input1.dim0 已由同一个 ShapeSymbol(H) 证明

Runtime:
  input0: [2, 8]
  input1: [16]
```

如果执行阶段只检查 `runtime_checks`，这个错误不会被发现，因为该约束在符号阶段已被判定为 `kSatisfied`，不会进入 `runtime_checks`。

因此，执行阶段需要两类验证：

```text
TensorSpec runtime validation
  -> 验证 kSatisfied 约束依赖的前提没有漂移

ShapeConstraint runtime validation
  -> 验证 kDeferred 约束在 concrete shape 下成立
```

## 二、威胁模型

从 Plan-time 到 Runtime，以下因素可能破坏已证明约束：

| 威胁 | 场景 | 影响 |
| --- | --- | --- |
| 输入形状漂移 | 同一 `ExecutionPlan` 被用于不兼容输入 | static dim、symbol identity、rank、dtype 前提失效 |
| 权重替换 | 热加载新模型权重，packed weight 指针失效 | kernel 输入数据与 plan 不一致 |
| Workspace 绑定错误 | arena 大小不足、offset 错误、并发复用错误 | workspace 越界或错位 |
| Kernel 注册表变更 | prepare 后 registry 被动态修改 | 已解析 kernel 与执行 kernel 不一致 |
| Operator 状态变异 | `Prepare()` 缓存状态被意外修改 | plan-time 缓存失效 |
| Tensor 生命周期错误 | `TensorView` 指向已释放内存 | runtime 数据指针悬垂 |

本文档当前优先解决第一类：**输入形状漂移导致已证明形状约束失效**。

## 三、推荐方案：符号感知 TensorSpec 运行时验证

### 3.1 原理

不要把运行时保障设计成一个泛用 graph solver，也不要再引入 `TensorSpecFingerprint` 中间层。`TensorSpec` 已经包含 dtype 和 `SymbolicShape`，而 `ShapeSymbol` 自身已经通过 `IsStatic()` / `IsUnknown()` / `IsSymbolic()` 编码了维度类别和值，因此运行时验证器应直接读取 `TensorSpec`。

```text
TensorSpec + per-Execute ShapeBindingContext
```

它的职责是验证 concrete runtime tensor 是否满足 plan-time `TensorSpec`：

- dtype 必须一致；
- rank 必须一致；
- static dim 必须等于 concrete dim；
- 同一个 generated `ShapeSymbol` 在一次 `Execute()` 中必须绑定到同一个 concrete dim；
- `ShapeSymbol::Unknown()` 不绑定、不传播 identity。

### 3.2 核心数据结构

执行阶段直接使用 `TensorSpec` 作为验证输入，不新增 fingerprint 类型。唯一需要新增的是 per-Execute 局部绑定表：

```cpp
struct ShapeBindingSource {
    size_t step_index;
    TensorPort port;
    size_t dim_index;
};

struct ShapeBinding {
    int64_t value;
    ShapeBindingSource source;
};

class ShapeBindingContext {
public:
    Status Bind(ShapeSymbol symbol,
                int64_t value,
                ShapeBindingSource source);

    StatusOr<int64_t> Lookup(ShapeSymbol symbol) const;

private:
    std::unordered_map<int64_t, ShapeBinding> bindings_;
};
```

`ShapeBindingContext` 不应暴露为新的图推导入口。它只是 runtime TensorSpec validator 的内部状态，用来保证同一个 symbol identity 在执行阶段没有绑定冲突。

### 3.3 绑定语义

| `ShapeSymbol` 类型 | Runtime 行为 |
| --- | --- |
| static dim `>= 0` | 不进入 map，直接检查 concrete dim 是否等于 static value |
| generated symbol `<= -2` | 第一次绑定记录 concrete value；后续绑定必须相同 |
| `Unknown(-1)` | 不绑定、不传播；只要求 concrete dim 合法 |

伪代码：

```cpp
Status ShapeBindingContext::Bind(ShapeSymbol symbol,
                                 int64_t value,
                                 ShapeBindingSource source) {
    if (value < 0) {
        return Status::InvalidArgument("runtime dimension must be non-negative");
    }

    if (symbol.IsStatic()) {
        if (symbol.GetStaticValue() != value) {
            return Status::InvalidArgument("static shape dimension mismatch");
        }
        return Status::Ok();
    }

    if (symbol.IsUnknown()) {
        return Status::Ok();
    }

    auto [it, inserted] = bindings_.emplace(symbol.value(), ShapeBinding{value, source});
    if (!inserted && it->second.value != value) {
        return Status::InvalidArgument("symbolic shape binding conflict");
    }
    return Status::Ok();
}
```

### 3.4 执行顺序与跨 step 传播

`LayerRunner::Run()` 为每次执行创建一个 `ShapeBindingContext`，并把它传给每个 step。这个 context 是 per-Execute 的，因此会跨 step 共享：

```text
LayerRunner::Run(plan, bindings):
  ShapeBindingContext shape_bindings;
  for each step:
    RunStep(step_index, step, bindings, shape_bindings)
```

这意味着 `ShapeBindingContext` 不只验证当前 step 的 tensor 是否满足 spec，还会把当前 step 中已经出现的 symbol 绑定传播给后续 step：

```text
Step 0 output_specs: output0 = [B, S]
Step 0 runtime output0: [1, 4096]
  -> bind S = 4096

Step 1 input_specs: input0 = [B, S]
Step 1 runtime input0: [1, 4096]
  -> lookup/bind S = 4096, pass

Step 1 runtime input0: [1, 8192]
  -> bind S = 8192 conflicts with existing S = 4096, fail
```

`RunStep()` 中的顺序应使用 `ValidateAndBind` 语义，而不是单纯校验：

```text
1. Bind workspace
2. 如果 step 有 input_specs/output_specs/runtime_checks，读取 StepTensorBinding
3. ValidateAndBind input_specs against TensorView inputs
4. ValidateAndBind output_specs against MutableTensorView outputs（如果输出已绑定）
5. ValidateShapeConstraints(runtime_checks)
6. op->Run()
```

建议 API：

```cpp
Status ValidateAndBindTensorSpec(const TensorSpec& spec,
                                 const TensorView& tensor,
                                 ShapeBindingContext& bindings,
                                 ShapeBindingSource source);

Status ValidateAndBindTensorSpec(const TensorSpec& spec,
                                 const MutableTensorView& tensor,
                                 ShapeBindingContext& bindings,
                                 ShapeBindingSource source);
```

如果 Step 1 的 input tensor 与 Step 0 的 output tensor 是同一个实际 tensor，该机制会自然验证一致性。如果它们不是同一个 tensor，但 `TensorSpec` 引用了同一个 `ShapeSymbol`，也会强制它们的 concrete dim 一致；这是正确行为，因为同一个 generated `ShapeSymbol` 本身就表达 runtime identity 约束。

需要注意的是，`ShapeBindingContext` 只记录 runtime 已经出现过的 tensor 维度对 symbol 的 concrete 绑定，不做图遍历、不推导新关系，也不替代 `ShapeConstraint` evaluator。它是执行期轻量 binding table，不是 graph solver。

跨 step 传播还有一个关键前提：不同 step 的 `TensorSpec` 必须让相同语义维度引用同一个 `ShapeSymbol`，也就是 `ShapeSymbol::value()` 相同。如果 Step 0 的 output dim 和 Step 1 的 input dim 虽然语义相同，但分别由独立的 `ShapeSymbol::Create()` 生成，那么 `ShapeBindingContext` 会把它们视为两个无关 symbol，无法自动传播绑定。

当前 `ExecutionPlanBuilder` 对每个 step 独立调用 `InferOutputShapes()`。如果上游 lowering 没有把前一步 `output_specs` 传给下一步 `input_specs`，那么相邻 step 的 symbol identity 通常不会共享。要让跨 step 传播真正生效，Builder 或更上层 graph lowering 需要增加 symbol unification 步骤：将前一步 `output_specs` 中的 `ShapeSymbol` 传递到后续消费该 tensor 的 `input_specs`。

如果暂不实现跨 step symbol unification，`ShapeBindingContext` 会退化为 per-step 验证语义：每个 step 只验证自己的 `input_specs` / `output_specs` 与当前 runtime tensor 是否一致；跨 step 一致性需要通过显式 `runtime_checks`（例如 `DimEqualConstraint`）或 tensor 连接层的 shape 传递机制保障。

因此 `ExecutionStep` 需要保留 plan-time 输入规格：

```cpp
struct ExecutionStep {
    std::vector<TensorSpec> input_specs{};
    std::vector<TensorSpec> output_specs{};
    std::vector<ShapeConstraint> runtime_checks{};
    // ... existing fields
};
```

`ExecutionPlanBuilder` 在构建 step 时从 `ExecutionPlanNodeSpec.input_specs` 复制/移动到 `ExecutionStep.input_specs`。

### 3.5 与 `runtime_checks` 的职责边界

两者是互补关系：

| 组件 | 负责内容 | 示例 |
| --- | --- | --- |
| TensorSpec runtime validator | 符号阶段已证明约束的前提 | 同一个 `H` 在多个 input dim 上必须绑定为相同值 |
| ShapeConstraint evaluator | 符号阶段无法证明的 deferred 约束 | 两个不同 symbol 在 runtime 必须相等 |

`runtime_checks` 不应重新检查所有已证明约束，否则会重复建模并引入两套 truth source。已证明约束由 TensorSpec validator 保障，deferred 约束由 `ShapeConstraint` evaluator 保障。

## 四、其他保障方案

### 4.1 Packed Weights 生命周期绑定

`ExecutionStep.packed_weights` 当前是裸指针，依赖外部 `ModelInstance` 或 mmap backing 生命周期。后续应改为结构性所有权保障，例如：

```cpp
struct ExecutionStep {
    std::shared_ptr<const void> packed_weights_holder;
    const void* packed_weights = nullptr;
};
```

如果 packed weights 来自 mmap，则 `ExecutionPlan` 可以持有 backing 对象的 `shared_ptr`，热路径继续使用裸指针快速访问。

这是结构性修复，不应受验证级别开关控制。

### 4.2 Workspace 边界防护

当前 `RuntimeBindingContext::BindWorkspace()` 已经检查 workspace arena 是否能满足 `WorkspaceRequirement`。后续可以增强 `WorkspaceBinding`：

```cpp
struct WorkspaceBinding {
    void* data = nullptr;
    size_t size = 0;
    size_t offset = 0;
    size_t arena_total = 0;

    AM_NODISCARD bool WithinArena() const noexcept {
        return offset <= arena_total && size <= arena_total - offset;
    }
};
```

Debug / Paranoid 模式下可以再考虑 guard page，但不作为当前阶段默认目标。

### 4.3 Kernel 密封

`Prepare()` 阶段已经把 resolved kernel 缓存在 operator 内部，普通执行不再查询 registry。因此 kernel registry 后续变化通常不会影响已构建 plan。

如需进一步防御 operator 内部状态被破坏，可以记录：

```cpp
struct KernelSeal {
    KernelFunc fn = nullptr;
    const char* debug_name = nullptr;
    size_t attrs_hash = 0;
};
```

该方案优先级低于 TensorSpec runtime validation、weights 生命周期和 workspace 边界。

### 4.4 类型状态编码

类型状态编码可以从编译时保证 `ValidateParams -> CheckInputSpecs -> InferOutputShapes -> Prepare -> Run` 顺序不可绕过，但需要重构 operator 生命周期类型。该方案适合服务化阶段或大规模 API 稳定化后实施，不建议作为当前 Phase 1 的第一步。

### 4.5 条件性验证策略

验证级别可以作为后续横切配置引入：

```cpp
enum class VerificationLevel : uint8_t {
    kNone,
    kMinimal,   // dtype + rank
    kStandard,  // TensorSpec runtime validation + runtime_checks
    kFull,      // Standard + workspace boundary + kernel seal
    kParanoid,  // Full + debug-only memory guards
};
```

当前建议先实现 `kStandard` 的语义，不急于引入配置复杂度。

## 五、实施路线

### Phase 1：当前闭环

1. `ExecutionStep` 增加 `input_specs`。
2. `ExecutionPlanBuilder` 保存 `ExecutionPlanNodeSpec.input_specs`。
3. 新增 TensorSpec runtime validator：
   - 验证 dtype；
   - 验证 rank；
   - 验证 static dim；
   - 用 per-Execute `ShapeBindingContext` 验证 generated symbol identity；
   - 忽略 `Unknown(-1)` 的 identity 绑定。
4. `LayerRunner::Run()` 创建 per-Execute `ShapeBindingContext`。
5. `LayerRunner::RunStep()` 在 `ValidateShapeConstraints(runtime_checks)` 之前调用 TensorSpec validator。
6. 保留当前 `ShapeConstraint` evaluator 作为 `kDeferred` 约束的 runtime 检查。

### Phase 2：结构性内存安全

1. packed weights 生命周期绑定。
2. workspace boundary 信息增强。
3. 必要时引入 `VerificationLevel`。

### Phase 3：长期架构强化

1. 类型状态编码。
2. kernel seal。
3. debug/paranoid guard page。

## 六、核心原则

1. **不要把 `runtime_checks` 扩展成“所有约束再检查一遍”**。它只负责 `kDeferred`。
2. **`kSatisfied` 的运行时保障来自 TensorSpec 前提验证**，尤其是 symbol identity 的 concrete binding 一致性。
3. **`ShapeSymbol::Unknown()` 不具备 identity 语义**，不能进入 binding map。
4. **ShapeBindingContext 是执行期临时状态，不是 graph solver**。
5. **weights 生命周期和类型状态属于结构性保障**，不应被 runtime verification level 关闭。
6. **先实现 `kStandard` 行为，再引入配置开关**，避免在机制尚未稳定前扩大 API 面。
